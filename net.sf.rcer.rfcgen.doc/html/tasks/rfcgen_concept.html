RFC Mapping Language Concepts

The SAP R/3 Connector SDK contains a tool to facilitate the task of performing RFC calls. Instead of writing the Java coding to perform the input and output mapping manually, you can generate the classes from a mapping definition that is written in a compact language. The SDK even contains a wizard that generates a basic mapping definition by examining the function module definition.

Before writing the mapping definition, you have to decide whether you want to use call-style mapping or request-response-style mapping. If you use call-style mapping, a single call class is generated. You prepare an instance of this class by setting the input parameters, then execute the call. The results of the function module execution are written into the instance of the call class. In other words, this mapping style works in-place. If you use request-response-style mapping, you have to use two classes. You prepare an instance of a request class - much the same way you would with a call class - and then execute the call. This does not change the request, but creates an instance of a response class instead. The following chapters contain detailed examples of both mapping styles.

To generate a basic mapping definition, make sure you're able to connect to a SAP R/3 system. Then, execute the following steps:

  1. Select File - New - Project and choose SAP R/3 Connector - RFC Wrapper Project.
  2. Remove the empty model.rfc from the source directory.
  3. Select File - Import and choose SAP R/3 Connector - Create RFC Mapping Definition.
  4. Enter one or more function modules you want to include into your mapping definition.
  5. Enter the name of the target Java package for the generated classes and choose the mapping style.
  6. Place the file as model.rfc into the source directory.
  7. Right-click on the generated workflow.oaw and choose Run as - oAW Workflow.

This will generate the mapping definition and the classes. Now you can examine the generated classes and, if necessary, change the mapping definition and repeat the last step as often as you like.

The generated classes support property change notification and can therefore be used in JFace data binding contexts without further adaption.

Note that you can use multiple mapping definitions to keep the definitions organized - just adapt the workflow definition accordingly
